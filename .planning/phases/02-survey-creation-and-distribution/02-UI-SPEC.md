## Spacing Scale

Declared values (multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, badge padding, inline spacing between text and icon |
| sm | 8px | Input internal padding (vertical), compact list item gap |
| md | 16px | Card padding, form field gap, button horizontal padding |
| lg | 24px | Section heading margin-bottom, modal content padding |
| xl | 32px | Card grid gap, page content top padding |
| 2xl | 48px | Page section breaks (e.g. between Survey List and Empty State) |
| 3xl | 64px | Page-level vertical padding (main content area top/bottom) |

Exceptions:
- File upload drop zone: minimum height 120px to meet touch/click target
- SMTP modal dialog: max-width 480px, internal padding 24px (lg)
- "Send Test Email" button touch target: minimum 44px height (accessibility)
- Status badge pill: 4px vertical / 8px horizontal padding (xs / sm)

## Typography

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Body | 14px | 400 (regular) | 1.5 | Form labels, table rows, modal body copy, helper text, status badges |
| Heading | 20px | 600 (semibold) | 1.2 | Page titles ("Surveys", "Settings"), modal headings, column headers |
| Display | 28px | 600 (semibold) | 1.15 | Empty state headlines only |

Declared weights: 400 (regular) and 600 (semibold) only. Status badges use weight 400; badge differentiation is handled by background color, not weight. Column headers use weight 600 (heading role, reduced to 12px via font-size utility).

Burmese overrides: same sizes and weights; font-family switches to Noto Sans Myanmar Variable via `.font-myanmar` class when locale is `my`. Line-height for Burmese body raises to 1.65 to accommodate Myanmar script descenders.

## Color

The codebase uses OKLCH CSS custom properties via shadcn. The sidebar implementation establishes Tailwind blue-600 as the functional accent. The neutral base-nova preset provides all surface and muted tokens.

| Role | Value | CSS Token | Usage |
|------|-------|-----------|-------|
| Dominant (60%) | #ffffff (white) | `--background` / `--card` | Page background, card surfaces, modal backgrounds |
| Secondary (30%) | oklch(0.985 0 0) ≈ #fafafa | `--sidebar` / `--muted` | Sidebar background, table row hover, input fill, card footer |
| Accent (10%) | #2563eb (blue-600) | Tailwind `blue-600` class | See reserved list below |
| Destructive | oklch(0.577 0.245 27.325) ≈ #dc2626 | `--destructive` | Delete survey confirmation only |

Accent (#2563eb / blue-600) reserved for:
1. Active sidebar nav item background (`bg-blue-50`) and text (`text-blue-700`)
2. Primary action buttons: "Create Survey", "Send Invitations", "Save Settings"
3. "Send Test Email" button (primary variant)
4. File upload drop zone border on drag-over state
5. Progress/upload indicator fill
6. Focus rings on all interactive elements (`focus-visible:ring-blue-500/50`)
7. Brand name in sidebar header (`text-blue-600`)

Accent is NOT used for: table borders, card borders, badges, helper text, secondary buttons.

Source: blue-600 established in `src/components/admin/AdminSidebar.tsx` (Phase 1 output). CSS tokens from `src/app/globals.css`.

## Visual Hierarchy

Eye entry point notes per surface — where the user's eye lands first and how attention flows.

### Survey List Page (`/admin/surveys`)
**Focal point:** The "New Survey" button (top-right, blue accent) is the primary focal point when surveys exist. It is the highest-contrast element on an otherwise neutral surface. Eye flow: button (top-right) → survey list rows (title column) → status badges (right column) → row actions ("View", "Delete").

When no surveys exist, the focal point shifts to the empty-state heading "No surveys yet" (28px display, center-aligned) followed immediately by the "Create Your First Survey" CTA below it.

### Survey Creation Form
**Focal point:** The "Survey Name" input at the top of the form — it is the first focusable element and carries the `<Label>` + required indicator. Eye flow: name field → description textarea → file upload drop zone → "Create Survey" CTA (bottom-right).

### Excel Import — Question Preview Table
**Focal point:** The import result alert banner (success or error) displayed above the table after parsing. Eye flow: alert banner → question count in banner → table rows (English text column, widest) → type badges (right).

### SMTP Settings Page
**Focal point:** The "Host" field at the top of the form — it is the first required field and determines connection viability. Eye flow: host → port → username → password → "Save Settings" + "Send Test Email" button row.

### SMTP Onboarding Modal
**Focal point:** The modal heading "Set Up Email Delivery" (20px semibold, top of dialog). Eye flow: heading → body explanation → "Configure Now" CTA (primary, blue) → "Skip for Now" (ghost, lower visual weight).

### Email Distribution Form
**Focal point:** The survey `<Select>` dropdown — it gates the send action and sits at the top. Eye flow: survey select → email textarea → parsed count display ("12 email addresses found") → "Send Invitations" CTA.

### Send Progress / Confirmation View
**Focal point:** The progress indicator or result summary card fills the center of the screen, replacing the form entirely. Eye flow: status text ("Sending 3 of 12..." or "12 invitations sent successfully") → result detail → "Send Another Batch" button.

## Interaction Contracts

### File Upload (Excel Import)
- Drop zone accepts `.xlsx` and `.xls` MIME types only; reject others immediately with error copy above
- On valid file drop or selection: show filename, file size, and lucide `FileSpreadsheet` icon in a row
- The remove button in the upload feedback row renders as `<button aria-label="Remove file">` containing the lucide `X` icon; no visible text label is required
- Upload state: replace drop zone with progress indicator (indeterminate spinner) while parsing with exceljs
- On success: show question preview table below; "Import Questions" confirmation button appears
- On failure: clear file selection, show inline error, re-enable drop zone
- The file is parsed client-side by the API route; no file content is stored — only extracted question records

### SMTP Onboarding Modal
- Trigger condition: admin navigates to `/admin/surveys/new` or `/admin/email` when no SMTP config exists in storage
- Modal is non-dismissible via ESC or backdrop click — only via "Configure Now" (routes to settings) or "Skip for Now" (closes and sets a session flag to not re-trigger until next login)
- SMTP modal must not block the settings page itself

### Test Email Send
- "Send Test Email" button is disabled until all 5 SMTP fields are non-empty
- On click: button enters loading state (spinner, label "Sending..."), disabled
- On success: button re-enables; inline green text "Test email sent to {from address}" appears for 5 seconds then fades
- On failure: button re-enables; inline red text with raw error code persists until user edits a field

### Send Invitations Progress
- "Send Invitations" button disabled until: survey selected AND at least 1 valid email address present
- On submit: replace form with progress view showing current count ("Sending 3 of 12...")
- Dispatch is sequential (not parallel) to avoid SMTP rate limiting — one email per tick
- On complete: show result summary card; "Send Another Batch" resets form to initial state

### Survey Status Badges
- Draft: `<Badge variant="secondary">` — gray/neutral
- Active: `<Badge>` with `bg-blue-50 text-blue-700` — matches sidebar active color
- Closed: `<Badge variant="outline">` — border only, muted text

## Bilingual Constraints

All UI strings must have entries in both `messages/en.json` and `messages/mm.json`. This is a hard requirement from UIUX-04.

| String Category | Notes |
|-----------------|-------|
| Form labels | English + Burmese; font-family switches via `.font-myanmar` on `lang="my"` body |
| Error messages | Must be translated — bilingual error messages are required for Burmese-locale admin usage |
| Status badges | "Draft" / "Active" / "Closed" → Burmese equivalents required |
| Email template copy | Admin-facing confirmation copy in English only (admin UI defaults to English); employee-facing email template must be bilingual (English header + Burmese header in same email, per EMAL-07) |
| SMTP modal | Burmese translation required |
| Empty states | Burmese translation required |
